# Аудит проекта GAMBA Tycoon
**Дата аудита:** 30 марта 2026 г.

## 1. Общая архитектура и структура проекта
Проект представляет собой монорепозиторий (`gamba-tycoon-monorepo`) с разделением на клиентскую (`casino`) и серверную (`server`) части. Запуск процесса разработки скриптом `npm run dev` из корня использует пакет `concurrently` для одновременного старта двух сред. 

Стек технологий:
- **База данных / Auth:** Firebase (Firestore, Firebase Auth)
- **Backend:** Node.js, Express
- **Frontend:** React 19, TypeScript, Vite, TailwindCSS v4, Radix UI, Framer Motion, Phaser 3.

---

## 2. Анализ Backend (папка `server`)
Серверная часть написана на чистом JavaScript (CommonJS). Это микросервис на базе Express для работы с критическими операциями в базе данных через Firebase Admin SDK.

**Положительные аспекты (Безопасность и Архитектура):**
- Использованы `helmet` для базовой защиты заголовков и `cors` для настройки доступа.
- Реализован Rate Limiting (`express-rate-limit`) для защиты от DDoS / спама запросами (30 запросов в 5 минут на IP).
- Валидация входных данных осуществляется с помощью строгих схем библиотеки `Zod`.
- Аутентификация: реализована кастомная middleware `authenticateToken`, проверяющая валидность `JWT ID Token` от Firebase.
- Транзакции в БД: банковские переводы происходят внутри `db.runTransaction()`, что предотвращает состояние гонки (race conditions) и некорректные списания. Создаются логи операций как для отправителя, так и для получателя.

**Найденные проблемы и риски:**
1. **Уязвимость в эндпоинте `/api/admin/deploy`**: Ииспользуется `child_process.exec('/root/deploy.sh')`. Хотя маршрут защищен ролевой моделью (`process.env.ADMIN_NICKNAME`), вызов системных скриптов напрямую из Node.js потенциально может привести к проблемам безопасности или зависаниям процесса (Особенно если в скрипт передавать не экранированные пользовательские данные, хотя сейчас таких нет).
2. Общий стиль кода: сервер не использует TypeScript (в отличие от фронтенда), архитектура не разбита на роуты или контроллеры. Весь функционал сосредоточен в монолитном файле `index.js`, что усложнит поддержку.

## 3. Анализ Frontend (папка `casino`)
Клиентская часть выполнена с использованием современных практик и фреймворков:
- **Структура**: Классическое и понятное разделение (components, contexts, games, hooks, lib, pages, types).
- **Стилизация**: Использование Tailwind v4 в паре с библиотекой компонентов Radix UI (что дает высокую доступность - a11y) и анимациями `framer-motion`.
- **Игровая логика**: В зависимостях указан Phaser 3 (`phaser`), что означает наличие canvas-основанных 2D игр/механик. 
- Настроены статические анализаторы (`eslint`, строгий `tsconfig`).

## 4. Итоги и рекомендации
Проект находится в хорошем состоянии, кодовая база фронтенда отличается свежими версиями библиотек и соблюдением современных стандартов.

**Что необходимо улучшить:**
1. **Рефакторинг backend**: Переписать серверную часть (`server`) на TypeScript и вынести логику из монолитного `index.js` в отдельные маршруты, контроллеры и сервисы (паттерн Controller-Service).
2. **Безопасность**: Заменить или ограничить механизмы деплоя из бэкенда. Вызов shell скриптов прямо из приложения — это антипаттерн (используйте CI/CD пайплайны, например GitHub Actions).
3. **Управление конфигурациями**: Проверьте, что ключи доступа (`serviceAccountKey.json`) добавлен в `.gitignore` во избежание утечек.
